## 4. Iconify category map — NEBM-oriented, safe fallback, established pattern

The keyword→glyph map (ordered specific-before-generic, so `polycarbonate`/`frp` win before the generic `sheet` rule) resolves each of the **12 seeded top-level names** re-confirmed present in `db.json` (`parentId:null`: WPC Louvers, Polycarbonate Sheets, FRP Sheets, Waterproofing Products, Tiles, Doors, Hardware, Plumbing, Bath Fittings, Cement, Steel Rods, Special Products):

## 7. Findings — `09b` claims reconciled; two new documentary nuances; no code defect

### 7.1 — RECONCILE (`09b` §6.3): the Special-Products gate is `false` because of the *seed*, not the code default
`09b` recorded the Discover section as "Featured / All Products with Special Products absent" because `useDealsConfig().enabled` was false. Re-grounding closes the loop on *why*: the code default is `enabled: true` ([DealsConfigContext.js:24](src/context/DealsConfigContext.js); [dealsConfig.js:46–47](src/utils/dealsConfig.js)), but the provider loads the admin config from `apiService.deals.getConfig()` ([DealsConfigContext.js:36–47, :62](src/context/DealsConfigContext.js)), and the seeded `db.json` carries **`"dealsConfig": { "enabled": false }`** ([db.json:5717–5719](db.json)). So the runtime gate is genuinely false out of the box — `09b`'s observation is accurate and explained by the seed, not a code default. The `quickLinks` memo filters `/special-offers` out when disabled ([SidebarMenu.js:70–73](src/components/SidebarMenu/SidebarMenu.js)), mirroring the Header. **Note the scope:** only the Discover *quick link* is gated — the Special Products *top-level category* (with its gold `.catParentIconGold` glyph) still renders in the category tree regardless. Expected behaviour; it reappears when the admin enables the Special Offers page.

### 7.2 — NEW NUANCE (defensible): two Discover tiles sit outside the two-colour brand palette
The §9 palette criterion is met where it matters — brand blue `#1885d8` on every category glyph / active state, gold `#fa9c4c` confined to the two Special-Products spots. But the other two Discover quick-link tiles use decorative off-palette tints: **Featured** → `.toneViolet` (`#8b5cf6`, [:420–423](src/components/SidebarMenu/SidebarMenu.module.css)) and **All Products** → `.toneIndigo` (`#6366f1`, [:410–413](src/components/SidebarMenu/SidebarMenu.module.css)). These are neither brand blue nor gold. This **reuses the drawer's established multi-tone icon-tile aesthetic** (the tones predate Prompt-09) and applies only to small decorative tiles, not primary/active colour — a defensible trade-off, not a break. Recorded so a later editor tightening the storefront to a strict blue+gold palette knows these two tiles are the remaining deviation.

### 7.3 — DOCUMENTARY NIT: a misleadingly-named gold class + a dead tone class
- `.tonePink` ([:430–433](src/components/SidebarMenu/SidebarMenu.module.css)) styles the **gold** Special-Products quick-link tile (`color: var(--sf-color-accent)` = `#fa9c4c`, `rgba(250,156,76,.13)` bg) — the class name is a leftover from the generic boilerplate and no longer describes its colour. Purely cosmetic naming drift; the rendered gold is correct.
- `.toneAmber` ([:425–428](src/components/SidebarMenu/SidebarMenu.module.css)) is **dead** — `grep toneAmber src/components/SidebarMenu/SidebarMenu.js` → 0 hits (it styled the retired "Best Sellers" link). Harmless leftover, as `09b` §6.4 recorded. Both are documentary nits for a future CSS tidy, not defects.

### 7.4 — NUANCE (spec-compliant, unchanged): Iconify glyphs resolve from the Iconify API at runtime
`@iconify/react` fetches `mdi:*` data from `api.iconify.design` on first use (then caches), so glyphs paint a frame or two after mount and, fully offline, would render blank tiles. This is the **established app pattern** (`WHY_CHOOSE_US`, §4) and the spec explicitly directed "@iconify/react (already bundled) … for premium category glyphs," so it is compliant. The name→icon *fallback* covers unseen names, not network-offline; a reviewer wanting offline-guaranteed glyphs would bundle the set (`@iconify-icons/mdi`) rather than treat blank tiles as a regression. As `09b` §6.2 recorded.

### 7.5 — CARRY-FORWARD (unchanged): `/orders` route for "My Enquiries"
The account "My Enquiries" still points at `/orders` — spec §4.7 says "route `/orders` for now"; repointing `/orders`→`/enquiries` is owned by the enquiry-flow/admin prompts (16–20 / 25 / 28), exactly as prior notes recorded. Not a Prompt-09 defect.
