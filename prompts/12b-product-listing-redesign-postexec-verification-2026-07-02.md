## 5. Findings — no defects; reconciliations + carry-forwards

### 5.1 — DECISION (§5 + §11): the merchandising badges are gated so Prompt-11's bands stay clean
Prompt-12 §5/§9 require Featured/Special/Trending **and** the honest discount badge on the listing card; Prompt-11 §5/§7 required **no "% off" urgency** on the curated homepage Featured/Special bands (which render the same shared card). Both hold via a single `showBadges` prop: **Special is always additive** (never gated — Prompt-11's rule), while **Featured/Trending/discount reveal only when the caller opts in**. The listing passes `showBadges`; the bands/carousels don't. Runtime confirms it: `/products` shows the full set, the homepage bands show only the 9 gold Special badges. Sanctioned reconciliation, not a defect — grounded in `prompts/00` §3/§5, which keeps NEBM's honest `comparePrice`-derived discounts while removing Buy-Now/coupons/deal-timers.

### 5.2 — DECISION (§7): the listing delegates BOTH grid and list to the shared card (no fork)
§7 offered "match the inline renderer **or** delegate to `<ProductCard>`." Per the prime directive (and `11c` §7.3, which anticipated the listing "adopting the shared card"), the fork was **deleted** and the listing now renders `<ProductCard>` for grid **and** list (via the new `layout` prop). One card, one source of truth — the badge + enquiry action can never drift between the listing and the rest of the storefront.

### 5.3 — SANCTIONED cross-surface effect: the canonical card's action is now icon-only everywhere
Prompt-11 §3 asked for an "'Add to Enquiry List' icon-button w/ tooltip" but its implementation still rendered an icon **+ visible "Add to Enquiry" text**. Prompt-12 §6 firmly requires **no visible text**. Making the canonical card icon-only therefore both fulfils Prompt-11's stated intent and satisfies Prompt-12, and — because the card is shared — it propagates uniformly to the homepage bands, `/special-offers` and the related/bundle carousels (runtime: 16 band buttons, none with text). Intended, consistent evolution of the one card, not a regression.

### 5.4 — CARRY-FORWARD (prompt 15): the card price is still the clean `sellingPrice`; the struck compare price is deferred
The card shows `PriceBlock price={sellingPrice} showSavings={false}` (no inline struck compare price); the honest **discount** is surfaced as the media chip. The struck-through MRP + inline "% off" the *old* listing showed inside the price row is owned by **prompt 15**'s `PriceBlock` card mode ("keep the honest discount badge on the media consistent with the card price"), along with `priceType` exact/tiered/onEnquiry. So between now and prompt 15 the listing conveys savings via the media discount chip rather than a struck price — a deliberate, sequenced division of labour, not a loss. `PriceBlock` itself is untouched.

### 5.5 — CARRY-FORWARD: `getProductMinPrice`'s discount ignores `priceType`
The media discount chip is driven purely by `comparePrice > price`, so a `tiered`/`onEnquiry` product that also carries a `comparePrice` could show a "% OFF" chip. This is harmless today (honest, derived from real numbers) and is subsumed by **prompt 15**, which makes the card price `priceType`-aware (a `tiered` card shows "From ₹X", an `onEnquiry` card shows "Price on Enquiry"); the chip can be reconciled with the mode there. Not a Prompt-12 defect.

### 5.6 — NOTE: the list view keeps its grid/list toggle; only the columns/skeleton CSS remain in Products.module.css
The forked inline-card CSS was removed; the file now only lays out the grid and the loading-skeleton shell (`.card`/`.cardImageWrap`/`.cardBody`), with the card visuals living in `ProductCard.module.css`. The grid/list **toggle** is preserved (list view = the shared card's `layout="list"` horizontal variant). No feature removed.
