Prompt-10 §4.3/§9 require the Featured/Special bands to use the **shared** `ProductCard` "with an 'Add to Enquiry List' icon-button" and §5/§9 forbid any "% off" urgency — but that card is the single canonical card, and its enquiry-button + badge changes are nominally prompts 11/16's and its price display prompt 15's. Re-grounding confirms the minimal "coordinate" work was done and nothing more: (a) "Add to Cart" → "Add to Enquiry" icon-button ([ProductCard.js:94–125](../src/components/storefront/ProductCard.js)); (b) the red "% OFF" `.discountBadge` removed (grep of `ProductCard.module.css` → 0 hits); (c) `comparePrice` no longer fed to `PriceBlock` (`price={sellingPrice} showSavings={false}`, [:92](../src/components/storefront/ProductCard.js)) → clean price. Left to their owners: the **gold "Special" badge** (prompt 11), the **priceType-aware display** (exact/tiered/on-enquiry, prompt 15), and the **cart toast copy** (prompt 16). `PriceBlock` is untouched (not in the diff), so the PDP still shows a compare price until prompts 14/15. Net-positive, enquiry-correct, no regression to any completed prompt — a reconciliation, not a defect.

### 7.2 — NEW DOCUMENTARY NIT: `Home.js` keeps a dead `styles.dark` reference (its `isDarkMode` is now vestigial)
`10b` §2 says `Home.module.css` "now follows `body.dark` automatically, no per-rule `.dark`." Re-grounding confirms that at source — the only `.dark` in `Home.module.css` is a *comment* ([:4](../src/pages/Home/Home.module.css)), with no `.dark` selector — **but** `Home.js` still applies `` `${styles.homePage} ${isDarkMode ? styles.dark : ""}` `` ([:181](../src/pages/Home/Home.js)). With the CSS rule gone, `styles.dark` resolves to `undefined`, so in dark mode the root className gains a harmless literal `"undefined"` token that matches nothing, and `useTheme()`/`isDarkMode` is now effectively dead weight in this file. (Contrast `CTASection.js`, which *kept* its `.dark.cta` rule ([CTASection.module.css:7](../src/components/CTASection/CTASection.module.css)), so its `isDarkMode` toggle is still live.) Purely a leftover for a future tidy — no visual or runtime effect. Recorded, not a defect.

### 7.3 — NEW MINOR NUANCE: category skeleton count (8) vs. 12 categories actually rendered

This is the **established app pattern** (the SidebarMenu and header do the same, per `09b`/`09c` §7.4) and the spec sanctions `@iconify/react`, so it is compliant. As `10b` §5.2 recorded.
